## Проблема которую решаем

У агента множество баз данных (context_memory, leviathan_kb, journal, claude_projects и др.)  
Но между ними нет связей — агент не знает куда идти за знаниями.  
Результат: либо сканирует всё (дорого по токенам), либо не использует вообще.

**Цель:** агент получил задачу → за 1-2 запроса нашёл нужное знание → не тратит контекст на лишнее.

---

## Ключевая идея: граф хранилищ

Не граф потоков выполнения (LangGraph), а **граф знаний с маршрутизацией**.

```
Задача
  ↓
RouteIndex.match(задача)
  → паттерн найден → (база, id, следующий_узел)
  → паттерн не найден → новая ветка
        ↓
   реальный поиск (PyPI / GitHub / KB)
        ↓
   сохранить новый паттерн → RouteIndex.add()
```

Со временем третий случай (новая ветка) становится редкостью — граф заполняется.

## Архитектура баз (узлы графа)

```
┌─────────────────────────────────────────────────────┐
│                  ROUTE INDEX                        │
│  pattern_hash | keywords | db_name | record_id      │
│  next_hop_db  | next_hop_id | mode | maturity       │
└──────────────────────┬──────────────────────────────┘
                       │ маршрутизация
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
  domains.db      patterns.db      risks.db
  (каталог         (паттерны        (риски по
   доменов)         задач)           паттернам)
       │
       ▼ для каждого домена
  ┌─────────────────────────────────┐
  │         ДОМЕН: e-commerce       │
  ├──────────┬──────────┬───────────┤
  │ arch.db  │designs.db│  code.db  │
  │архитект. │UI шаблоны│ модули кода│
  └──────────┴──────────┴───────────┘
```

## Структура баз

| База | Что хранит | Ключ поиска |
|------|-----------|-------------|
| `route_index.db` | маршруты паттерн→база→id | keywords, hash |
| `domains.db` | список доменов и их дочерние базы | keywords |
| `patterns.db` | паттерны задач → домен + варианты | embedding/keywords |
| `arch.db` | архитектурные решения | домен + сложность |
| `designs.db` | UI шаблоны и дизайн-решения | домен + платформа |
| `code.db` | код-модули с вариантами | домен + функция + вариант |
| `risks.db` | риски паттернов по режимам | паттерн + mode |

## Вариативность внутри паттерна

Каждый паттерн хранит **несколько вариантов** с контекстом выбора:

```json
{
  "pattern": "корзина покупок",
  "domain": "e-commerce",
  "variants": [
    {"id": "a", "name": "session-based", "complexity": "low",  "risk": 1, "when": "MVP, нет авторизации"},
    {"id": "b", "name": "SQLite",        "complexity": "mid",  "risk": 2, "when": "один пользователь"},
    {"id": "c", "name": "Redis+queue",   "complexity": "high", "risk": 4, "when": "нагрузка > 100 rps"}
  ],
  "mode": ["personal", "freelancer", "public"],
  "maturity": "tested"
}
```

Агент смотрит контекст задачи (нагрузка, стек, режим) и выбирает нужный вариант.

---

## Режимы (mode) и риски

Режим определяется **сервисом** (портом/конфигом), не задачей:

| Режим | Порт | Риск-планка | Описание |
|-------|------|-------------|----------|
| `personal` | 8200 | мягкая (до 14) | лаборатория, для себя |
| `freelancer` | 8200 | средняя (до 9) | ArbitrCockpit, клиентские задачи |
| `public` | 82XX | жёсткая (до 4) | Book Factory, Diet Platform, пользователи |

**Ключевое правило:** если PUBLIC не может выполнить задачу по рискам —  
агент решает её в PERSONAL, сохраняет внутри, наружу не отдаёт.  
Паттерн помечается `maturity: experimental` и ждёт адаптации.

### Матрица рисков

**Технические (одинаковы для всех режимов):**
| Фактор | Вес |
|--------|-----|
| subprocess / shell | +3 |
| Сетевые запросы | +2 |
| Рекурсия без лимита | +2 |
| Внешняя библиотека | +1 |
| Запись на диск | +1 |

**Юридические/этические (зависят от режима):**
| Фактор | personal | freelancer | public |
|--------|----------|------------|--------|
| Торренты / P2P | +1 | +5 | +8 |
| Скрейпинг | +1 | +3 | +5 |
| Персональные данные | +2 | +5 | +9 |
| Обход ограничений | +2 | +6 | +9 |
| Авторские права | +1 | +4 | +8 |

**Шкала:**
- 0–4 → 🟢 LOW — используем
- 5–9 → 🟡 MEDIUM — адаптируем
- 10–14 → 🔴 HIGH — ищем альтернативу
- 15+ → ⛔ BLOCK — отказываем, объясняем

## SolutionEngine — инструмент агента

Отдельный tool call. Агент вызывает явно когда нужно решить задачу оптимально.

```
ШАГ 1: RouteIndex.match(задача)
  → нашли паттерн в KB → берём вариант по контексту → ГОТОВО
  → не нашли → идём дальше

ШАГ 2: Реальный поиск (только для новых паттернов)
  → PyPI search
  → GitHub code search  
  → собственная генерация (2 варианта)
  → итого 4 варианта

ШАГ 3: Матрица рисков
  → считаем риск каждого варианта по mode агента
  → отсекаем BLOCK

ШАГ 4: Выбор
  → минимальный риск
  → если равны → предпочитаем своё (контролируем код)

ШАГ 5: Адаптация
  → таймауты (15с)
  → обработка ошибок
  → комментарии

ШАГ 6: Сохранение в KB
  → новый паттерн → route_index.db
  → полный лог (почему отказались от остальных)
  → mode + maturity
```

### Инспирация: AutoTTS (май 2026)

Исследователи UMD/Google/Meta дали агенту искать алгоритмы самому вместо написания их вручную.  
Результат: -70% токенов при той же точности, $40 за весь поиск.  
Ключевые уроки применённые здесь:
- **Офлайн-среда** = KnowledgeBase с историей → не вызываем LLM заново
- **Итеративность** = агент видит логи прошлых попыток, не только результат
- **Один контроллер** = mode (personal/freelancer/public) управляет всеми весами
- **Полные логи** = сохраняем почему отказались, не только что выбрали

## Maturity pipeline паттернов

```
experimental → tested → production
     │              │           │
  PERSONAL      FREELANCER   PUBLIC
  решили сами   проверили    зрелый
                на клиенте   паттерн
```

Агент сам повышает maturity когда паттерн успешно применялся N раз.

## Итог

Граф знаний — это то что превращает набор баз данных в **самообучающуюся память агента**.  
Каждая задача либо находит готовый путь (дёшево, детерминировано),  
либо строит новый и запоминает его для следующего раза.  
Со временем агент всё реже обращается к LLM за базовыми решениями — он уже знает.
